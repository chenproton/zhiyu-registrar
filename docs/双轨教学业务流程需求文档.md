# 数字教务平台 — 双轨教学业务流程需求文档

> **版本**: v1.0  
> **日期**: 2026-06-01  
> **适用范围**: 学历体系（课程教学）+ 能力体系（场景教学）双轨并行的职业教育教务管理

---

## 一、业务背景与核心命题

### 1.1 双轨并行的必要性

职业教育面临**学历体系**与**能力体系**的结构性分离：

| 维度 | 学历体系（传统课程） | 能力体系（场景教学） |
|------|---------------------|---------------------|
| **目标** | 知识传授、学分积累、学位获取 | 岗位胜任力、技能认证、就业 readiness |
| **单元** | 课程（Course） | 场景/任务（Scene/Task） |
| **评价** | 考试分数、平时成绩、GPA | 能力等级、项目成果、企业评价 |
| **师资** | 学校教师主讲 | 学校教师 + 企业导师协同 |
| **场地** | 教室、机房、实验室 | 实训基地、企业现场、虚拟仿真环境 |
| **进度来源** | 课程平台学习数据 | 场景平台任务完成数据 |
| **归档物** | 成绩单、学历证书 | 能力档案袋、项目作品集 |

**核心命题**：两套体系不能各自为政，必须在同一教务平台中实现：
1. **统一排课视图** — 教室排课与实训排课不冲突
2. **统一备课入口** — 教师在一个界面完成课程备课和场景任务设计
3. **统一成绩出口** — 学分成绩与能力等级在学生档案中汇聚
4. **统一师资调度** — 企业导师与学校教师的协同排程

### 1.2 教师角色的重新定义

在传统教学中，教师是**知识传授者**。在场景教学中，教师升级为：

- **场景任务设计师** — 将岗位真实任务拆解为教学子任务
- **企业协调员** — 邀请企业导师、协调导师排程、管理导师评价
- **能力映射师** — 将子任务与能力点、评价量规进行关联
- **双平台管理员** — 同时在课程平台和场景平台维护教学资源

系统必须支持这种角色升级，而非简单叠加工作量。

## 三、分阶段详细需求

### 阶段一：人培方案输入 → 教学大纲 → 教学计划（规划层）

#### 3.1.1 人培方案管理

**功能定位**：人才培养方案是全部教学活动的源头，决定了一个专业"培养什么人、用什么课程培养"。

**核心功能**：
- **8 大板块结构化编辑**：基本信息、职业面向、培养规格、课程体系、学时学分、师资队伍、教学条件、质量保障
- **AI 辅助导入**：支持 Excel/Word 格式的历史人培方案解析，自动提取课程列表、学时分配、培养目标
- **版本管理**：每次修改生成新版本，保留历史对比
- **审批流程**：系部提交 → 学院审核 → 教务处备案 → 发布执行

**与双轨的关联**：
- 人培方案中的课程体系同时包含**理论课程**、**实践课程**、**场景课程**三种类型
- 场景课程在课程体系中以特殊标记（🧪 图标 + 紫色标签）展示
- 场景课程的能力目标需关联到岗位能力模型

#### 3.1.2 教学大纲生成

**触发条件**：人培方案审批通过后，自动或手动生成对应课程的教学大纲。

**生成逻辑**：
| 课程类型 | 大纲模板差异 |
|---------|-------------|
| 理论课程 | 侧重知识点体系、理论学时分配、考核方式（考试/考查） |
| 实践课程 | 侧重实验/实训项目清单、操作规范、安全要求 |
| 场景课程 | **新增**：岗位任务映射、子任务链、能力点对应、企业导师参与说明、评价量规 |

**场景课程大纲特有字段**：
- `scenePlatformId`：关联场景平台课程 ID
- `subTaskDefinitions[]`：预定义子任务名称、能力点、建议工位、建议时长
- `mentorParticipationMode`：企业导师参与方式（全程/部分/远程）
- `evaluationRubricId`：评价量规模板 ID
- `safetyRequirements`：安全操作规范

#### 3.1.3 教学计划生成

**触发条件**：教学大纲审批通过后，按学期生成教学计划。

**双轨视图**：

**视图 A — 学历体系（课程维度）**：
```
学期 1：程序设计基础(4学分/64学时) | 高等数学(4学分/64学时) | ...
学期 2：数据结构(4学分/64学时) | 软件工程实践(场景/4学分/64学时) | ...
```

**视图 B — 能力体系（岗位→场景维度）**：
```
岗位：Web前端开发工程师
  └── 场景 1：企业官网开发 (对应课程：Web前端综合实训)
  └── 场景 2：电商平台前端 (对应课程：前端框架应用)
岗位：后端开发工程师
  └── 场景 3：API接口设计与测试 (对应课程：软件工程实践)
```

**系统实现**：
- 教学计划表格增加"类型"列（理论/实践/场景），场景行紫色高亮
- 统计卡片增加"场景课程"计数
- 教学计划详情页展示课程→场景的映射关系

### 阶段二：课程分配 → 排课中心（调度层）

#### 3.2.1 课程分配（任课绑定）

**功能定位**：将教学计划中的课程/场景绑定到具体班级和教师。

**传统课程分配**：
- 选择班级 → 选择课程（从课程池）→ 选择主讲教师 → 设置周课时/周次模式
- 支持分层教学（A层/B层分班，不同教师授课）

**场景课程分配的特殊处理**：
- 选择班级 → 选择场景课程 → **双教师绑定**：学校教师（主责）+ 企业导师（协责）
- 企业导师从师资库中筛选（`isEnterpriseMentor = true`）
- 子任务预分配：系统根据大纲预定义，建议各子任务的负责教师和导师
- 工位预分配：根据班级人数和场景需求，推荐实训基地/工位区域

#### 3.2.2 排课中心（核心冲突解决层）

**业务挑战**：传统排课只需考虑「教室+教师+时间」三维冲突；场景排课还需考虑「工位+企业导师时间+设备可用性」额外维度。

**统一排课引擎设计**：

```
输入：
  - 传统课程：{班级, 课程, 教师, 周次, 星期, 节次, 教室需求}
  - 场景任务：{班级, 场景, 教师, 导师, 周次, 星期, 节次, 工位需求, 设备需求}

冲突检测规则：
  1. 教室/工位在同一时段只能被一个任务占用
  2. 学校教师在同一时段只能出现在一个教室/工位
  3. 企业导师在同一时段只能参与一个场景任务
  4. 班级在同一时段只能上一门课/做一个场景任务
  5. 设备在同一时段只能被一个场景任务使用（如数控机床）
  6. 场景任务的连续节次要求（通常需要 3-4 节连排）

输出：
  - 无冲突的课表
  - 冲突告警列表（人工调整）
```

**4 Tab 设计**：

| Tab | 传统课程 | 场景任务 |
|-----|---------|---------|
| **班级课表** | 按班级展示周课表，理论课蓝色标记 | 场景任务紫色标记，显示导师姓名 |
| **课程管理** | 课程列表，查看各班级排课情况 | 场景列表，查看子任务排课分布 |
| **教学计划** | 按学期分配课时到班级 | 按学期分配场景任务到班级 |
| **任务编排** | 课程→任务的自动转换 | 场景→子任务的编排，导师时间窗口匹配 |

**场景排课的特殊 UI**：
- 场景任务卡片显示：🧪 场景名称 + 学校教师 + 企业导师 + 工位编号
- 导师时间窗口可视化：用甘特图展示导师可用时段
- 设备占用状态：实时显示设备当前是否可用

### 阶段三：备课中心（准备层）

#### 3.3.1 核心理念：同一教师、同一入口、不同模板

教师不应在两个系统中分别备课。系统根据任务类型（`traditional` vs `scene`）自动切换备课模板。

#### 3.3.2 传统课程备课模板

**3 阶段结构**：

| 阶段 | 核心内容 | 检查项 |
|-----|---------|-------|
| **课前准备** | 教学目标（知识/能力/素养三维）、导学方案、课前资源（课件/视频/预习题） | 目标已设定、资源已上传 |
| **课堂实施** | 课堂活动设计、课件资源、讨论话题/随堂测 | 活动已设计、课件已准备 |
| **课后拓展** | 课后作业、拓展资源、课后测验 | 作业已布置 |

**课程特有信息**：
- 课程平台 ID 绑定（用于进度同步）
- 知识点映射（课程知识点 ←→ 大纲知识点）
- 大纲章节映射（第 N 次课 ←→ 第 M 章节）

#### 3.3.3 场景教学备课模板

**3 阶段结构**（表层与传统相同，底层逻辑不同）：

| 阶段 | 传统课程 | 场景教学 |
|-----|---------|---------|
| **课前准备** | 教学目标（知识/能力/素养） | **任务目标**（能力点映射 + 产出物定义） |
| | 课前资源（课件/视频） | **场景导入资源**（案例文档/角色卡/环境配置指南） |
| | 导学方案 | **角色设定方案**（学生扮演什么岗位角色） |
| **课堂实施** | 课堂活动（讲授/讨论/演示） | **子任务链执行**（任务 1 → 任务 2 → 任务 3） |
| | 课件资源 | **场景平台链接** + **开发环境配置** |
| | 讨论话题/随堂测 | **测评要点**（能力评价维度） |
| **课后拓展** | 课后作业 | **复盘要求**（能力自评 + 改进计划） |
| | 拓展资源 | **成果提交规范**（代码/文档/演示视频） |

**场景特有信息面板**（备课页面上方固定展示）：

**子任务状态管理**：
- 计划中 → 进行中 → 已完成（教师可在任务执行阶段手动更新）
- 每个子任务关联独立的进度追踪

#### 3.3.4 备课检查引擎

**传统课程检查清单**（7 项）：
1. 已设定教学目标
2. 课前资源已准备
3. 课堂活动已设计
4. 课后作业/拓展已布置
5. 课程平台已关联
6. 知识点映射已完成
7. 大纲章节已映射

**场景教学检查清单**（10 项）：
1. 已设定教学目标
2. 课前资源已准备
3. 课堂活动已设计
4. 课后作业/拓展已布置
5. 已关联实践场景平台
6. 子任务已拆分（至少 1 项）
7. 每个子任务有描述
8. 子任务能力点已映射
9. 评价量规已配置
10. 工位/设备已配置
11. 安全要求已设定
12. 企业导师排程已安排

**检查通过 → 备课状态变为 "completed" → 关联任务状态可流转至 "ready"**

---

### 阶段四：开课中心（发布层）

#### 3.4.1 开课检查引擎

开课是教学正式面向学生的起点，必须通过强制性检查。

**传统课程开课检查**（4 项）：
1. ✅ 备课已完成
2. ✅ 场地已确认
3. ✅ 学生已通知
4. ✅ 教学资料已准备

**场景教学开课检查**（6 项）：
1. ✅ 备课已完成
2. ✅ 场地/实训基地已确认
3. ✅ 学生已通知
4. ✅ 教学资料已准备
5. ✅ 工位/设备已检查（系统向场地管理模块查询设备状态）
6. ✅ 企业导师已确认（导师收到排程通知并确认参与）

**检查 UI**：
- 弹窗展示所有检查项，通过的显示绿色勾选，未通过的显示橙色警告
- 进度条实时展示通过率
- 未全部通过时，"确认开课"按钮禁用
- 场景课程额外展示导师确认状态和设备检查状态

#### 3.4.2 开课后的状态联动

```
开课确认 → 课程/场景任务状态从 "ready" → "published"
         → 学生端课表/任务列表可见
         → 如果是场景任务，同步通知场景平台创建任务实例
         → 企业导师收到参与提醒（按排程时间提前通知）
```

---

### 阶段五：任务执行与课堂管理（执行层）

#### 3.5.1 传统课程执行

- **课前**：学生通过课程平台预习，完成课前测试
- **课中**：教师按备课方案授课，系统记录签到、课堂活动参与
- **课后**：学生提交作业，系统记录平台学习进度
- **成绩采集**：平时成绩（平台数据）+ 期中/期末考试成绩

#### 3.5.2 场景任务执行

场景教学不是"上一节课"，而是"完成一个任务链"。执行流程：

```
课前（线上）：
  └── 学生接收角色卡、阅读场景背景、预习相关知识点
  └── 教师在场景平台发布任务说明和环境配置指南

课中（线下/混合）：
  └── 子任务 1：环境配置（导师现场/远程指导，教师巡视）
  └── 子任务 2：核心任务执行（学生分组在工位实操）
  └── 子任务 3：成果整合与初步测试
  └── 每个子任务完成后，学生在场景平台提交阶段性产出

课后（线上）：
  └── 学生完善成果、提交最终作品
  └── 教师 + 企业导师共同评审（按评价量规打分）
  └── 学生完成能力自评和复盘反思
```

**系统支持**：
- **课堂签到**：支持普通签到 + 实训基地位置签到
- **课堂记录**：教师记录每个子任务的实际执行情况、学生表现、设备问题
- **进度同步**：从场景平台拉取子任务完成率，实时更新任务进度
- **导师参与记录**：记录企业导师实际到场时间、指导内容、评价意见

#### 3.5.3 课表执行与调课

**标准课表导出**：
- 支持按班级/教师/场地导出周课表
- 传统课程和场景任务在同一课表中展示，场景任务紫色标记
- 导出格式：PDF / Excel / 图片

**调课管理**：
- 传统调课：变更教室/时间/教师
- 场景调课：变更工位/时间/教师/导师（导师变更需重新确认）
- 调课申请触发冲突检测，通过后自动通知相关方

---

### 阶段六：学业认定（归档层）

#### 3.6.1 成绩认定（学历体系）

- 成绩来源：课程平台（传统）/ 场景平台（场景）
- 场景课程的成绩构成可展示子任务成绩明细
- 成绩异常检测：场景任务未完成但成绩已录入 → 系统告警

#### 3.6.2 能力认定（能力体系）

- 能力档案自动汇聚场景任务的完成情况和评价结果
- 场景能力达标统计：学生完成多少个场景、每个场景的能力等级
- 企业导师评价纳入能力认定来源

#### 3.6.3 双轨汇聚

学生在毕业时同时拥有：
- **学历证书** + **成绩单**（学历体系产出）
- **能力档案袋** + **项目作品集**（能力体系产出）

两者在系统中通过学生 ID 关联，在「学生详情」页可一键查看双轨全景。

```
        │

        │                                                                     │

        │
        └── 场景课程 ──→ SceneSyllabus (场景大纲)
                              │
                              ├── subTaskDefinitions: [SceneSubTaskDefinition]

                                       │

```

### 4.2 关键字段设计

**Task（学习任务）—— 双轨统一实体**：
```typescript
interface Task {
  id: string
  type: 'traditional' | 'scene'          // 类型决定后续流程分支
  status: 'draft' | 'ready' | 'published' | 'in_progress' | 'evaluating' | 'completed' | 'archived'
  
  // 共同字段
  courseName: string
  classId: string
  facultyId: string
  dayOfWeek: number
  periods: string[]
  weeks: string
  venueId: string
  
  // 场景特有
  enterpriseMentorId?: string
  workStationId?: string
  externalPlatformId?: string        // 场景平台任务 ID
  externalPlatformType?: 'course' | 'scene'
  
  // 子任务（仅 scene 类型有效）
  sceneSubTasks?: SceneSubTask[]
  
  // 进度（来源根据 type 自动区分）
  progressSummary?: TaskProgressSummary   // dataSource: 'course_platform' | 'scene_platform' | 'mixed'
}
```

**PreparationTask（备课任务）—— 双轨统一入口**：
```typescript
interface PreparationTask {
  id: string
  taskType: 'traditional' | 'scene'    // 决定展示哪个备课面板
  
  // 三阶段备课（共同结构）
  stages: { pre: {...}, in: {...}, post: {...} }
  
  // 场景备课特有
  scenePrep?: {
    externalPlatformId?: string
    subTaskPreparations: [{
      subTaskId: string
      subTaskName: string
      description?: string
      abilityPointIds: string[]
      evaluationRubricId: string
      workstationId?: string
      equipmentChecklist: string[]
      safetyRequirements: string
      enterpriseMentorNotes: string
      mentorSchedule?: {
        participationType: 'onsite' | 'remote' | 'async'
        timeSlot: string
      }
    }]
  }
  
  // 课程备课特有
  coursePrep?: {
    externalPlatformId?: string
    knowledgePointIds: string[]
    syllabusChapterMapping: [...]
  }
}
```

## 五、教师端工作流设计

### 5.1 传统课程教师的系统操作流

```
周一：
  └── 登录系统 → 查看本周课表
  └── 点击「备课」→ 编辑课前目标/资源/导学方案 → 保存
  └── 系统自动检查备课完整性 → 显示「备课已完成」

周三上课：
  └── 课前 15 分钟：系统推送「今日课程提醒」
  └── 课堂中：使用备课中的课件和活动设计
  └── 课后：记录课堂小结、标记签到情况

周末：
  └── 查看课程平台同步的学生学习进度
  └── 批改作业、录入平时成绩
```

### 5.2 场景教学教师的系统操作流

```
学期初（准备阶段）：
  └── 登录系统 → 查看分配给自己的场景教学任务
  └── 点击「备课」→ 进入场景备课面板
  └── 按子任务逐一配置：
        ├── 子任务 1：描述 + 能力点 + 工位 + 设备 + 安全要求
        ├── 子任务 2：...
        └── 企业导师排程：选择导师 → 设置参与类型 → 设置时间窗口
  └── 系统自动检查（10 项）→ 全部通过 → 「备课已完成」

学期中（执行阶段）：
  └── 每次场景课前：
        ├── 检查系统通知：导师是否已确认参与
        ├── 检查设备状态：场地管理模块显示设备是否可用
        └── 系统推送「开课检查」→ 6 项全部通过 → 确认开课
  
  └── 场景课中：
        ├── 启动子任务 1 → 学生开始执行
        ├── 导师按排程时间到场（现场/远程）
        ├── 教师巡视各工位，记录学生表现
        └── 子任务 1 完成 → 学生在场景平台提交产出 → 标记「进行中」→「已完成」
  
  └── 场景课后：
        ├── 查看场景平台同步的子任务完成率
        ├── 教师 + 导师共同按评价量规打分
        └── 学生提交复盘反思

学期末（总结阶段）：
  └── 查看全班场景任务完成情况统计
  └── 批量录入/确认场景成绩
  └── 生成学生能力档案评价
```

### 5.3 企业导师的系统操作流

```
收到排程邀请（邮件/短信/系统通知）：
  └── 点击确认参与 / 申请调整时间
  
场景课当日：
  └── 签到（现场扫码 / 远程连线确认）
  └── 按备课中的「导师备注」重点指导学生
  └── 在系统中填写指导记录和评价意见
  
课后：
  └── 查看学生提交的阶段性产出
  └── 按评价量规打分（与学校教师协同）
```

## 六、系统兼容性设计

### 6.1 向后兼容：纯传统教学的学校

如果学校暂不开通场景教学，系统表现为：
- 教学计划中只有理论/实践课程，无场景课程
- 备课页面只展示传统课程模板
- 排课中心只展示班级课表和课程管理
- 开课检查只有 4 项
- 教师感受不到场景功能的存在

### 6.2 向前兼容：逐步引入场景教学

学校可按专业/年级逐步引入场景教学：
- 在「人培方案」中，将部分实践课程标记为「场景课程」
- 系统自动为该课程启用场景备课模板和场景排课逻辑
- 其他课程保持传统模式不变
- 教师只在被分配场景任务时，才看到场景功能

### 6.3 双轨并行：同一教师同时负责传统和场景

- 教师的「我的任务」列表中，传统任务和场景任务混合展示
- 传统任务用 📘 蓝色标记，场景任务用 🧪 紫色标记
- 点击不同任务，自动进入对应类型的备课/执行界面
- 教师的课表中，传统课程和场景任务在同一周课表中展示

## 七、非功能性需求

### 7.1 性能要求
- 排课冲突检测：1000 条任务量下 < 2 秒
- 课表导出：单班级周课表 < 1 秒
- 进度同步：场景平台数据拉取 < 5 秒

### 7.2 可靠性要求
- 开课检查引擎为强制性关卡，不可绕过
- 备课状态变更记录审计日志
- 导师排程变更需双方确认，避免单方面修改

### 7.3 扩展性要求
- 场景平台接口预留标准化对接位置（externalPlatformId + webhook）
- 评价量规支持自定义扩展
- 新的场景类型可通过配置新增，无需改代码

## 九、待决策事项

1. **场景平台对接**：是否与特定厂商（如用友、华为）的场景平台做深度对接，还是保持 mock 级别？
2. **导师费用结算**：企业导师的课时费/指导费是否在系统中管理？
3. **场景课程学分认定**：场景任务完成的学分是否与理论课程学分等值？
4. **跨专业场景共享**：同一场景任务是否可分配给多个专业的班级？
5. **虚拟仿真场景**：纯线上虚拟场景（如 VR/AR 实训）是否纳入场景教学管理？
